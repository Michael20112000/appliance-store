## Purge (D-09)

| # | Крок | Pass / Fail |
|---|------|-------------|
| 1 | Зупинити залежність від каталогу/замовлень (опційно закрити зайві вкладки) | |
| 2 | Виконати: `CONFIRM_DB_PURGE=yes npm run db:purge` | |
| 3 | Переконатися, що скрипт завершився без помилки; у логах видно видалення Message → … → Category | |
| 4 | **Auth збережено:** користувачі адмінки все ще можуть увійти | |

### Що purge **не** видаляє (D-08)

- `StorePhone`, `StoreEmail`, `StoreAddress`, `CallbackRequest`
- Таблиці Better Auth: User, Session, Account, Verification
- Файли в Cloudinary (можуть залишитися «сирі» assets — P2)

Після purge **футер може показувати телефон/email** — це не провал empty-state тесту.

## Маршрути після purge (D-18)

Відкривати кожен URL у браузері. Критерій: **немає HTTP 5xx**, сторінка рендериться з очікуваним порожнім текстом.

| # | URL | Очікування | Pass / Fail |
|---|-----|------------|-------------|
| 1 | `/` | 200; **немає** секції `#kategorii` / блоку «Категорії» | |
| 2 | `/katalog` | 200; каталог без товарів — OK | |
| 3 | `/katalog/[slug]` | Якщо категорій немає після purge — **404**; якщо категорія лишилась порожньою — «У цій категорії поки немає товарів» | |
| 4 | `/admin/kategorii` | «Немає категорій» або порожня таблиця | |
| 5 | `/admin/tovary` | «Товарів не знайдено…» (заклик створити товар) | |
| 6 | `/koszyk` (гість, без логіну) | Порожній кошик гостя; **не** редірект на «Увійти» | |
| 7 | `/admin` (дашборд) | Віджет замовлень: «Замовлень ще немає» | |
