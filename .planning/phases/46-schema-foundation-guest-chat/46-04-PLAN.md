  <action>
    The current `getPusherClient()` creates a singleton Pusher instance. The problem: the singleton is created once and cannot know the guestToken at creation time (guestToken is state in ChatProvider). pusher-js supports `channelAuthorization.params` as a function that returns an object — use this to dynamically inject guestToken.

    Update `getPusherClient(key, cluster)` to NOT create a singleton when guestToken needs to vary. Instead:

    Option A (preferred — simpler): Add a new exported function `configurePusherGuestToken(token: string | null)` that updates a module-level variable `let currentGuestToken: string | null = null`. In the `channelAuthorization` config, add `params: () => currentGuestToken ? { guestToken: currentGuestToken } : {}`.

    If pusher-js does not support `params` as a function (check the installed pusher-js 8.5.0 types in node_modules), use a static approach: destroy the existing client and recreate it with the new params when the guestToken changes. Do not add the token to the channel name.

    Specifically: Add to `pusher-client.ts`:
    - `let currentGuestToken: string | null = null;`
    - `export function setGuestTokenForPusher(token: string | null): void { currentGuestToken = token; }`
    - In `getPusherClient()` channelAuthorization config: add `params: { guestToken: currentGuestToken ?? "" }` (static at client creation time — caller sets the token before `getPusherClient()` is first invoked)

    If the static approach is used (params set at creation time), ChatProvider must call `setGuestTokenForPusher(guestToken)` before the Pusher subscription effect runs. Add this call at the start of the Pusher useEffect in chat-provider.tsx: `setGuestTokenForPusher(guestToken)`.

    IMPORTANT: Read the actual pusher-js types in `node_modules/pusher-js/types/src/core/options.d.ts` or similar to understand whether `params` accepts a function or static object before implementing. Choose the approach that matches the actual API.

    If neither approach is feasible from the API, fall back to passing guestToken as a custom header: `channelAuthorization: { endpoint: "/api/chat/pusher/auth", transport: "ajax", headers: { "x-guest-token": guestToken } }` — and update parseAuthBody in the auth route to also read `request.headers.get("x-guest-token")`.

    Record the chosen approach clearly in the task SUMMARY for the reviewer.
