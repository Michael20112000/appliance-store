<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → POST /api/chat/new | Body contains optional guestToken — validated as UUID before service call |
| Client → POST /api/chat/claim | Authenticated session required; guestToken validated as UUID; userId sourced from session only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-47-claim-auth | Elevation of Privilege | POST /api/chat/claim | mitigate | Session check at route entry — `if (!session?.user) return 401` before any service call |
| T-47-claim-spoof | Spoofing | POST /api/chat/claim — guestToken | mitigate | `z.string().uuid()` rejects non-UUID tokens; `claimGuestConversation` uses `updateMany({ where: { guestToken } })` — only rows matching the token are updated; userId always comes from session (attacker cannot inject a different userId) |
| T-47-new-guest | Spoofing | POST /api/chat/new — guest guestToken | mitigate | `z.string().uuid()` validates token format; guest can only create new conversations for their own guestToken; no cross-user data exposure possible |
| T-47-claim-race | Tampering | POST /api/chat/claim — concurrent calls | accept | updateMany is idempotent: second call finds 0 matching rows (guestToken already null) and updates nothing; no duplicate claims possible |
| T-47-SC | Tampering | npm/pip/cargo installs | accept | No new packages; all pre-audited |
</threat_model>
