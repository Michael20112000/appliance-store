
    CHANGE 3 — Guest claim effect:
    After all existing useEffect blocks (but before the provider return), add:
    ```
    const claimAttemptedRef = useRef(false);

    useEffect(() => {
      if (!hasSession || claimAttemptedRef.current) return;
      if (typeof window === "undefined") return;

      const stored = localStorage.getItem(GUEST_CHAT_TOKEN_KEY);
      if (!stored) return;

      claimAttemptedRef.current = true;
      void (async () => {
        try {
          const res = await fetch("/api/chat/claim", {
            method: "POST",
            headers: { "Content-Type": "application/json" },
            body: JSON.stringify({ guestToken: stored }),
          });
          if (res.ok) {
            localStorage.removeItem(GUEST_CHAT_TOKEN_KEY);
            setGuestToken(null);
            setGuestTokenForPusher(null);
          }
        } catch {
          // claim failure is non-fatal — guest messages still visible via guestToken until page reload
        }
      })();
    }, [hasSession]); // eslint-disable-line react-hooks/exhaustive-deps
    ```
    Use `GUEST_CHAT_TOKEN_KEY` (the constant already defined in this file — do not hardcode the string). Use `setGuestToken` (the internal React state setter — not from context). Use `setGuestTokenForPusher` (already imported from "@/lib/pusher-client").

    Per RESEARCH.md Pitfall 7: `claimAttemptedRef.current = true` is set before the async call, not after, to prevent double-fire in React StrictMode.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → POST /api/chat/new | Client sends optional guestToken; server validates UUID format |
| Browser → POST /api/chat/claim | Client sends guestToken from localStorage; server requires authenticated session before processing |
| Pusher channel → ChatProvider | conversation:closed event arrives from Pusher; handler validates conversationId matches current context |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-47-pusher-spoof | Tampering | conversation:closed Pusher handler | mitigate | Handler checks `payload.conversationId !== conversationId` — events for other conversations are silently ignored; Pusher private channels require server-side auth before subscription, so event injection is not possible from untrusted clients |
| T-47-claim-double | Tampering | claimAttemptedRef in ChatProvider | accept | ref prevents double-fire but even if claim fires twice, updateMany is idempotent — second call finds 0 rows; no data integrity risk |
| T-47-banner-guestToken | Information Disclosure | ArchivedChatBanner POST /api/chat/new | accept | guestToken sent only if present in context (user's own token from localStorage); server validates UUID format; no cross-user disclosure possible |
| T-47-SC | Tampering | npm/pip/cargo installs | accept | No new packages; all pre-audited |
</threat_model>
